# majutsu v0.6.0 release evidence

## release

- version: 0.6.0
- date: 2026-06-26 JST
- version output: `mj 0.6.0`

## summary

- Changed the S3/GCS remote layout to use a human-readable environment label as
  the top-level host prefix, for example `vagrant/`, `mba22/`, and `winvr/`.
- Kept the host UUID inside metadata/head/GC records as the stable identity, but
  stopped using the UUID as the remote object path boundary.
- Removed the old shared `hosts/` / registry-style layout from the current
  remote implementation. The new layout intentionally has no fallback to the old
  layout.
- Normalized the production validation backend from the older
  `vagrant/<uuid>-encrypted` configured prefix to `majutsu-encrypted`.
- Verified the shared GCS backend with three hosts under one bucket:

```text
gs://majutsu-vagrant-winvr-s21g-twpro-20260608/
└── majutsu-encrypted/
    ├── mba22/
    ├── vagrant/
    └── winvr/
```

## migration note

Existing 0.5.x remotes using the old host-id path layout should be migrated by
configuring a new remote root, syncing each host into it, validating
`mj remote check`, and only then deleting the old remote prefix.

For this release candidate, the validation environment was migrated by
re-syncing vagrant, mba22, and winvr into:

```text
s3://majutsu-vagrant-winvr-s21g-twpro-20260608/majutsu-encrypted
```

After validation, the old `vagrant/c071...-encrypted` prefix and stale
`vagrant/perf-build*` test prefixes were deleted.
