# majutsu v0.6.8 release evidence

## release

- version: 0.6.8
- date: 2026-07-01 JST
- version output: `mj 0.6.8`

## summary

- Fixed `mj restore plan` on roots containing privileged live target paths such
  as `/etc/sudoers.d`.
- `restore plan`, `restore apply`, and `restore prepare` now detect permission
  denied during live target scans and avoid returning an incomplete plan.
- On an interactive terminal, mj asks for confirmation and re-runs itself via
  `sudo -E` using the absolute `current_exe()` path, so it does not depend on
  `sudo` finding `mj` through `secure_path`.
- On non-interactive output, mj reports that elevated privileges are required
  instead of silently skipping unreadable paths.

## validation

```sh
cargo fmt --check
cargo test --locked -q --test remote_file restore_plan_reports_permission_denied_live_delete_scan_entries
cargo check --locked -q
cargo test --locked -q --bin mj
cargo clippy --workspace --all-targets --locked -q -- -D warnings
scripts/sync-internal-crates.sh --check
scripts/package-release.sh
scripts/publish-crates-io.sh
```

Manual validation:

```sh
sudo mj -h
sudo env PATH="$PATH" mj -h
mj restore plan
```

- `sudo mj -h` can fail when sudo `secure_path` excludes
  `/home/vagrant/.cargo/bin`.
- `sudo env PATH="$PATH" mj -h` succeeds, confirming the issue is PATH lookup,
  not root execution.
- Non-TTY `mj restore plan` now returns a clear elevated-privilege error rather
  than an incomplete plan.
- A temporary unreadable root was tested through a PTY; answering `y` re-ran
  the command through `sudo -E <current_exe> --home <state> restore plan` and
  printed the plan successfully.

- result: pass
- coverage: fmt, clippy, binary unit tests, permission-denied restore plan
  regression, release package smoke, and crates.io dry-run.
- note: Podman MinIO was not run for this patch release because the change is
  local restore planning and privilege escalation behavior.
